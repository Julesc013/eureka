# Rust Local Index Parity Plan

Rust Local Index Parity Planning v0 defines the acceptance lane for a future
Rust local-index candidate. This is planning only: Rust Local Index parity
implementation is not started, Rust remains unwired, and Python remains the
oracle.

Python remains the oracle. Rust Local Index parity implementation is not started.

## Purpose

The Python Local Index now projects more than search rows. It carries source
records, resolved objects, states/releases, representations, members,
synthetic members, evidence records, article/scan records, compatibility
evidence, result lanes, and user-cost annotations. A future Rust candidate must
match that shape before it can be considered for any promotion.

This plan exists so the implementation milestone has a narrow target instead
of discovering the parity surface while writing Rust code.

## Future Rust Candidate Location

No Rust local-index implementation is added by this milestone.

After review, a future candidate may choose one of these bounded locations:

- a small `local_index` module inside `crates/eureka-core/`, if the candidate
  is only a pure record/projection parity model; or
- a separate Rust index crate, if SQLite execution or storage behavior is
  intentionally added in a later milestone.

Either option must remain isolated. Runtime, web, CLI, HTTP API, workers,
gateway, public-alpha paths, and surfaces must not call the Rust local index
until a later reviewed promotion explicitly changes that boundary.

## Allowed Divergences

Allowed divergences are governed by
`tests/parity/ALLOWED_DIVERGENCES.md#future-rust-local-index-divergences`.

Potential future allowances must be explicit and reviewed:

- FTS5 availability differences after normalization
- SQLite implementation or version metadata differences after normalization
- internal row IDs if public output remains stable
- non-public timing or performance differences

## Forbidden Divergences

These are not allowed:

- missing record kinds
- missing source IDs or source families
- missing synthetic members
- missing parent/member lineage
- missing article/scan records where Python has them
- missing compatibility evidence where Python has it
- missing result lanes or user-cost fields
- unstable query-result ordering after normalization
- private path leakage
- source placeholder overclaiming
- changed Python local-index behavior without refreshed oracle review

## FTS5 And Fallback Handling

Python prefers SQLite FTS5 when the local SQLite build supports it and falls
back to deterministic `LIKE` search when FTS5 is unavailable. Python-oracle
goldens normalize `fts_mode` because local SQLite feature availability can
vary.

Future Rust must document the selected SQLite/search path and normalize any
non-semantic feature metadata before comparison. Search result order must be
deterministic after normalization regardless of FTS availability.

## SQLite Caveats

The Python runtime stores the bounded index in SQLite. A future Rust candidate
must not assume a specific host SQLite build beyond what its parity report
records. Differences in tokenizer availability, SQLite compile options, or
non-public query-plan details are not parity failures if public normalized
outputs match and the divergence is recorded.

## Deterministic Ordering

Python query ordering is based on user cost, lower-cased label, and
`index_record_id`. Future Rust output must preserve stable public ordering
after normalization. Unstable ordering is not an allowed divergence.

## Path And Privacy Requirements

Future Rust parity output must not leak private absolute paths. Local index
paths, temporary roots, timestamps, and environment-specific SQLite details
must be normalized in parity outputs. Local/private placeholder sources must
remain placeholder or future-only and must not be overclaimed as indexed public
sources.

## No Runtime Wiring

Rust remains unwired. This planning milestone does not add a Rust local index,
does not add SQLite/indexing behavior, does not change Python runtime behavior,
and does not change web, CLI, HTTP API, worker, gateway, public-alpha, or
surface behavior.

## Acceptance Criteria

Rust Local Index Parity Candidate v0 may start only after this plan is reviewed
and these conditions are true:

- `python scripts/validate_rust_local_index_parity_plan.py` passes.
- `python scripts/validate_rust_local_index_parity_plan.py --json` emits a
  passing machine-readable summary.
- Current Python-oracle local-index goldens pass
  `python scripts/generate_python_oracle_golden.py --check`.
- Current source-registry parity catch-up checks still pass or honestly report
  Cargo unavailable.
- The future candidate has a reviewed decision about pure projection parity
  versus SQLite execution parity.
- Cargo availability requirements are explicit for the implementation
  milestone.

## Future Implementation Sequence

1. Review this plan and decide the candidate location.
2. Extend Python-oracle local-index goldens for the planned query cases.
3. Add a Rust record/projection model that preserves current Python output
   fields without SQLite execution.
4. Add a parity report emitter and normalize environment-specific fields.
5. Decide whether SQLite execution parity belongs in the same candidate or a
   later candidate.
6. Run optional Cargo checks where available.
7. Keep Rust isolated until a future promotion milestone explicitly changes
   runtime boundaries.

## Non-Goals

- no Rust local-index implementation
- no Rust SQLite/indexing behavior
- no Python local-index semantic change
- no runtime, web, CLI, HTTP API, worker, gateway, public-alpha, or surface
  wiring
- no live probes
- no external APIs
- no arbitrary local filesystem ingestion
- no production Rust backend claim
